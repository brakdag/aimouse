# Documentation Policy: Project Plans

## Overview
This document defines the standards and requirements for all planning documents within the `/docs/plans/` directory. The primary goal is to provide high-fidelity, unambiguous, and professional context to Large Language Models (LLMs) to ensure optimal code generation and architectural reasoning.

## Language Standard
**All documents must be written in native-level, professional English.**
*   **Rationale:** To minimize linguistic ambiguity and leverage the highest quality training data available to LLMs, thereby reducing the risk of "error bias" often associated with non-native or student-level documentation.
*   **Tone:** Technical, precise, and objective.

## Documentation Style
We follow a style inspired by Google's technical documentation standards:
*   **Clarity:** Use simple, direct sentences. Avoid jargon unless strictly necessary and defined.
*   **Structure:** Use hierarchical headings (H1, H2, H3) to organize thoughts.
*   **Conciseness:** Provide sufficient detail without verbosity. Every sentence should add value.
*   **Formatting:** Use Markdown for all structural elements (lists, tables, code blocks, bold text).

## Plan Document Structure
Every new plan document should follow this template:
1.  **Title:** Clear and descriptive.
2.  **Status:** (e.g., Draft, In Review, Approved, Superseded).
3.  **Objective:** A high-level summary of what this plan intends to achieve.
4.  **Context/Background:** Why is this being planned? What problem does it solve?
5.  **Proposed Solution:** Detailed technical breakdown.
6.  **Implementation Steps:** A sequential list of tasks.
7.  **Risks & Mitigations:** Potential pitfalls and how to handle them.
8.  **Success Criteria:** How do we know the plan worked?

## Maintenance
Plans are living documents. They should be updated as the project evolves, and superseded plans should be marked accordingly.
